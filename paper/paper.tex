\title{anvl: Composable Code Transformations for R}


\author{by Sebastian Fischer, Daniel Falbel, Thomas Kalinowski, Niko German, Martin Binder, and Bernd Bischl}

\maketitle

\abstract{%
The \CRANpkg{anvl} package offers a composable code transformation framework that allows users to express numerical algorithms efficiently in pure R. All code transformations are based on first tracing R code into a computational graph (an intermediate representation) via abstract evaluation. This allows combining jit-compilable operations with standard R code, the latter being ``staged out'' during compilation. Code transformations are implemented as functions that transform a computational graph into another computational graph. Except for the backends, the package is primarily written in R. Because of this and its modular design, \CRANpkg{anvl} is easy to extend and contribute to, e.g.~for adding new primitives or code transformations.
}

\subsection{Introduction}\label{introduction}

All statistical and machine learning models fundamentally rely on some form of vector or array operations.
Even though R as an interpreted language has performance limitations, relying on optimized matrix operations can often be fast enough for many use cases.
However, there are cases where the performance is not sufficient and one needs to resort to compiled languages such as C++.
Furthermore, one might additionally want to have access to GPU acceleration or automatic differentiation (AD) for gradient-based optimization.

To address this, we introduce \CRANpkg{anvl}, which is a composable code transformation framework for the R language.
R is no stranger to such transformations: base R already ships \texttt{Vectorize()}, which lifts a scalar function to a vectorized one and is similar in spirit to batch-map transformations such as \texttt{vmap} in JAX and PyTorch.
Its primary goal is to offer a set of primitive operations that can be jit-compiled, differentiated, and executed on different hardware backends.
A second, equally important goal is to provide an API that is well-suited to working with higher-dimensional arrays, so that users can focus on expressing \emph{what} an algorithm computes rather than on the bookkeeping of \emph{how} it is implemented in terms of R's array semantics.
It therefore allows users to implement numerical algorithms efficiently in pure R.
Two different execution backends are supported: an OpenXLA (``Open Accelerated Linear Algebra'') backend and an experimental Fortran backend.
The OpenXLA project is a compiler ecosystem used across different array computing frameworks.
Part of this project are the StableHLO language, which is a strongly typed language for representing array programs, and the PJRT (Pluggable Jit RunTime) library, which provides a common runtime for executing compiled programs.
Components of \CRANpkg{anvl} are the \CRANpkg{stablehlo} and \CRANpkg{pjrt} packages.
The former allows users to easily create StableHLO programs in R, while the latter interfaces the PJRT C library and handles compilation, data allocation, and execution.
Besides the OpenXLA backend, there is also an alternative, experimental backend that transpiles R to Fortran code via the \CRANpkg{quickr} package.
Because the \CRANpkg{anvl} package is primarily written in R, it is easy to extend and contribute to.

The closest related work is the JAX python library, from which various design elements are inspired or adopted \citep{ref-jax}.
Primarily, we are using a similar functional design and handle jit compilation via OpenXLA.
Also other design elements such as the handling of nested data structures or the type promotion rules are adopted from JAX.
We also want to highlight the Autodidax tutorial, which describes the core ideas underpinning JAX (todo: cite), as well as microjax, which is a minimalistic Python library demonstrating JAX's code transformation engine (todo: cite).
For the design of the \CRANpkg{stablehlo} package and the tracing implementation in \CRANpkg{anvl}, we have followed the design from \CRANpkg{mlr3torch}, which offers a similar graph-building interface for creating neural networks.
The API functions for the array operations are partly adopted from NumPy, which is a popular library for numerical computing in Python \citep{ref-harris2020array}.
For the design of the \CRANpkg{pjrt} package, we have partly borrowed from the gopjrt library \citep{ref-gopjrt} which offers a go interface to PJRT.

The OpenXLA project \citep{ref-openxla} is an open-source compiler ecosystem for machine learning, originally developed at Google as part of TensorFlow and later released as a standalone project.
One of its goals is to decouple numerical computing frameworks (the frontends) from hardware-specific implementations (the backends) by defining two standardized interfaces.
One of these is the StableHLO language \citep{ref-stablehlo}, a portable intermediate representation for array programs.
StableHLO is built on MLIR (Multi-Level Intermediate Representation), which is itself part of the LLVM compiler infrastructure.
MLIR provides a framework for defining and transforming intermediate representations at multiple levels of abstraction that are transformed into each other via passes that e.g.~optimize the program or lower the abstraction level.
The second interface is PJRT (Pluggable JIT RunTime), which defines a common API for the essential operations of a JIT-compiled array computing framework such as buffer management, compilation, and execution.
PJRT follows a plugin architecture, where each vendor can provide a shared library that implements the PJRT interface for their specific hardware.
This means that any framework that emits StableHLO and uses PJRT for execution can automatically target any hardware for which a PJRT plugin exists, including CPUs, NVIDIA GPUs, and Google TPUs.
Currently, the main compiler consuming StableHLO programs is the XLA compiler, but alternatives such as the IREE compiler \citep{ref-iree} also exist.

The main motivation behind developing \CRANpkg{anvl} is to go beyond the limitations of the \CRANpkg{torch} package \citep{ref-rtorch}, which is a deep learning framework for the R language.
It is based on LibTorch, which is the C++ library underpinning PyTorch \citep{ref-pytorch}.
The \CRANpkg{torch} package offers a collection of highly optimized array operations, as well as various modules for building and training neural networks.
As such, it also has support for backward-mode automatic differentiation and can be run on different hardware backends, including CPU and GPU.
Thus, its general capabilities are similar to those of \CRANpkg{anvl}, although it is more mature and contains more features.
However, \CRANpkg{torch} is primarily designed around so-called ``eager'' execution.
For example, the forward function of a neural network is simply an R function that repeatedly calls into LibTorch kernels; i.e., there is no compilation step.
\CRANpkg{anvl} supports both modes: by default, primitive operations execute eagerly just like in \CRANpkg{torch}, and JIT compilation is opted into by wrapping a function in \texttt{jit()}.
While eager execution has advantages regarding usability and expressiveness, it comes with performance implications.
For large neural networks that are in the primary focus of modern deep learning toolboxes, these performance implications might be less problematic, as the runtime is dominated by individually expensive operations such as the attention mechanism.
However, when models are smaller---as is not unusual for statistical models---JIT compilation can be especially attractive.
While torch offers a way to JIT-compile code via tracing code and converting it to TorchScript, this feature is in maintenance mode.
Furthermore, it is primarily designed for deployment and has, for example, no support for automatic differentiation.

Besides \CRANpkg{torch}, other R packages exist that offer GPU acceleration and automatic differentiation.
These are \CRANpkg{tensorflow} and \CRANpkg{keras3}, two deep learning frameworks that interface the TensorFlow Python runtime via \CRANpkg{reticulate}.
It is also possible to use Python's JAX library via \CRANpkg{reticulate} \citep{ref-jax}.
All of these packages also rely on OpenXLA for execution.
That XLA compilation is a valuable tool for implementing statistical algorithms is further evidenced by the popularity of JAX-based libraries such as NumPyro \citep{ref-numpyro} and BlackJAX \citep{ref-blackjax}, which drive much of the Bayesian modeling and inference ecosystem in Python.
While they provide access to more mature frameworks than \CRANpkg{anvl}, they require managing the Python dependency, as well as type conversions between R and Python.
Furthermore, with \CRANpkg{anvl} our goal is to develop an array computing library that is and will be developed primarily with the needs of the R community in mind.

Beyond those popular deep learning libraries, other R packages exist that offer hardware acceleration, automatic differentiation, jit compilation, or a combination of these.
We here primarily focus on those packages that are either available from standard repositories such as CRAN or Bioconductor, or which seem to be ready to be used.

A recent package that enables JIT compilation in R is the \CRANpkg{quickr} package, which transpiles R code into Fortran code and then compiles it \citep{ref-quickr}.
There are various differences between \CRANpkg{anvl} and \CRANpkg{quickr}.
First, \CRANpkg{quickr} has no support for automatic differentiation and currently only runs on CPU.
Besides, \CRANpkg{quickr} does not establish a new API, but instead aims to mirror the functionality of base R as closely as possible and act as a drop-in replacement for a subset of R code.
\CRANpkg{anvl}, in contrast, deliberately introduces a dedicated array-computing API that is designed to make working with higher-dimensional arrays more ergonomic and to separate \emph{what} an algorithm computes from \emph{how} it would otherwise have to be expressed in terms of R's array semantics.
This works because it is implemented via a static analysis approach, so the package does not need control over the functions that can be compiled, as is the case for the tracing approach followed in \CRANpkg{anvl}.
This also means that \CRANpkg{quickr} does not allow mixing ``non-compilable'' R code with compilable code, as is possible in \CRANpkg{anvl}, where the ``non-compilable'' code is staged out during tracing to the intermediate representation.
The tracing approach is more flexible, but also requires the user to be more careful about the code that they are writing.
There is also the ast2ast package, which translates a subset of R code into C++ \citep{ref-ast2ast}.
It is similar in spirit to \CRANpkg{quickr}, but also supports forward and backward-mode automatic differentiation.
Then, there are \CRANpkg{odin} and \CRANpkg{rxode2}, both of which focus on differential equations and compile a small subset of R code via translation to C, similar to \CRANpkg{quickr}.
However, they do not support GPU acceleration and their set of supported operations is more limited than \CRANpkg{anvl}'s.
There is another category of packages that offer special DSLs for statistical models which are compiled.
For example, \CRANpkg{brms} \citep{ref-brms} allows users to define Bayesian models via the familiar formula interface and translates them into Stan models which are compiled \citep{ref-stan}.
Stan itself is a probabilistic programming language and also has GPU support via OpenCL \citep{ref-opencl}.
It is also possible to directly use Stan in R, e.g.~via the \CRANpkg{rstan} package, but this requires writing the program itself in Stan.
There also exists \CRANpkg{nimble}, which is primarily intended for fitting hierarchical statistical models and works by generating C++ code and compiling it.
It also supports compiling custom R functions by generating C++ code, as well as forward and backward-mode automatic differentiation via CppAD \citep{ref-cppad}.
However, there is no support for GPU acceleration.
Then, there is also the \CRANpkg{compiler} package of the R language itself \citep{ref-r}.
It is used by the R interpreter to compile arbitrary R code into byte-code that is executed by the R virtual machine.
However, because it has to handle arbitrary R code, the possible optimizations are limited due to R's design \citep{ref-evaluating-r}.
This generally highlights the trade-off between expressiveness and performance.
In \CRANpkg{anvl}, we are focusing on a rather narrow set of numerical operations, which thus allows for aggressive optimizations and highly efficient execution.

While various R packages offer GPU acceleration for specific operations (e.g., \CRANpkg{xgboost}), generic GPU-accelerated array computing frameworks beyond the deep learning libraries from earlier are rare.
OpenCL is an open standard for heterogeneous parallel computing across CPUs, GPUs, and other accelerators, in which kernels are written in a C-like language and JIT-compiled by the device's vendor runtime.
One noteworthy mention is the \CRANpkg{OpenCL} package \citep{ref-r-opencl}, which exposes a low-level interface to OpenCL and requires users to hand-write and dispatch kernels themselves, offering no JIT compilation of R code or automatic differentiation.
It also leaves the OpenCL runtime setup to the user, who must install a vendor ICD separately, whereas \CRANpkg{anvl} ships prebuilt PJRT plugins that bundle the required runtime libraries.

Besides the packages already listed above, other R libraries exist with automatic differentiation capabilities.
For example, forward mode automatic differentiation via dual numbers is supported by the packages \CRANpkg{nabla}, \CRANpkg{salad}, \CRANpkg{dual}, and \CRANpkg{madness} \citep[\citet{ref-salad}, \citet{ref-dual}, \citet{ref-madness}]{ref-nabla}.
None of these packages have JIT compilation capabilities or support GPU execution.

There is also the \CRANpkg{RTMB} package which allows for forward and backward-mode automatic differentiation and interfaces the TMB C++ library (\citep{ref-tmb}).

Despite many packages offering some subset of \CRANpkg{anvl}'s features, none of the native R packages simultaneously offer GPU acceleration, JIT compilation, and backward-mode automatic differentiation.

\subsection{Background}\label{background}

\subsubsection{The OpenXLA Compiler Ecosystem}\label{the-openxla-compiler-ecosystem}

The OpenXLA compiler ecosystem consists of various individual components, but we will only cover the StableHLO intermediate language and the PJRT library which are used directly in \CRANpkg{anvl}.
One of the main goals of the OpenXLA project is to establish standardized interfaces between numerical computing frameworks such as JAX or anvl and their underlying execution engines.
This is achieved via the following two components:

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  The ``Pluggable Jit RunTime'' (PJRT) which defines a common interface for execution engines.
\item
  The StableHLO language, which is the intermediate representation produced by the frontend.
\end{enumerate}

If a numerical computing framework produces StableHLO programs and handles the execution via PJRT, it can be run on any hardware for which a PJRT plugin exists.
This turns an \(n \times m\) problem (\(n\) frontends need to implement their execution for \(m\) different hardware backends) into a \(n + m\) problem.

\paragraph{StableHLO}\label{stablehlo}

An example for the StableHLO intermediate representation is given below.
The main program takes in two \texttt{f32} arrays of shape \texttt{2x2}, multiplies them, and returns the result.

\begin{verbatim}
func @main(%arg0: tensor<2x2xf32>, %arg1: tensor<2x2xf32>) -> tensor<2x2xf32> {
  %0 = "stablehlo.multiply"(%arg0, %arg1) : (tensor<2x2xf32>, tensor<2x2xf32>) -> tensor<2x2xf32>
  return %0 : tensor<2x2xf32>
}
\end{verbatim}

It is expressed in MLIR (Multi-Level Intermediate Representation).
MLIR is part of the LLVM compiler infrastructure and provides a framework to define intermediate representations at multiple levels, also called dialects, that are transformed into each other via so-called passes.
Such passes can e.g.~perform optimizations or lower the abstraction level.
It is in single static assignment (SSA) form, meaning that each variable is assigned to exactly once, and is fully functional.
For a description of the language, see the \href{https://openxla.org/stablehlo/spec}{StableHLO specification}, which also contains a description of the around 100 supported primitive operations.
In order to use the StableHLO language in \CRANpkg{anvl}, it needs to be possible to translate R code into StableHLO programs.
For this purpose, we have created the \CRANpkg{stablehlo} package.
The package essentially consists of four parts:

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  An S3 class representation of the StableHLO language, which closely follows the specification.
\item
  A \texttt{repr()} generic that converts the class representation into a string representation that can be compiled by the XLA compiler.
\item
  A set of type inference functions for the primitive operations.
\item
  A builder API for easily building up StableHLO programs.
\end{enumerate}

The package is implemented fully in R, instead of interfacing the C++ implementation, which makes it easy to install for end users.
However, the \CRANpkg{stablehlo} package is only a partial implementation of the language and does not support all features.
For example, there is no support for complex numbers, a few primitive operations are missing, and there is currently no support for shape dynamism (unknown dimensions).

The example below illustrates how to use the builder API to re-create the StableHLO program from above.

\begin{verbatim}
library(stablehlo)
func <- local_func("main")
func
\end{verbatim}

\begin{verbatim}
#> func.func @main () ->  {
#> 
#> }
\end{verbatim}

\begin{verbatim}
x <- hlo_input("x", "f32", shape = c(2, 2), func = func)
x
\end{verbatim}

\begin{verbatim}
#> Variable %x in:
#> func.func @main (%x: tensor<2x2xf32>) ->  {
#> 
#> }
\end{verbatim}

\begin{verbatim}
y <- hlo_input("y", "f32", shape = c(2, 2), func = func)
z <- hlo_multiply(x, y)
f <- hlo_return(z)
f
\end{verbatim}

\begin{verbatim}
#> func.func @main (%x: tensor<2x2xf32>, %y: tensor<2x2xf32>) -> tensor<2x2xf32> {
#> %0 = stablehlo.multiply %x, %y : tensor<2x2xf32>
#> return %0 : tensor<2x2xf32>
#> }
\end{verbatim}

\begin{verbatim}
hlo_string <- repr(f)
\end{verbatim}

What happens in the background is that first, a new \texttt{Func} object is created with the name ``main'', which has reference semantics.
Then, \texttt{hlo\_input()} creates two \texttt{FuncInput} objects with the names ``x'' and ``y'' and specifies their type.
These are added to the \texttt{\$inputs} field of the built-up \texttt{Func} object.
The return values are \texttt{FuncValue} objects, which represent variables in the underlying StableHLO program that is being built up.
Objects of type \texttt{FuncValue} represent a specific value in a StableHLO program and are the core data structure underpinning the builder API.
Note that the R variable name \texttt{x} that is bound to the \texttt{FuncValue} does not need to coincide with the identifier ``x'' of the value in the StableHLO program.
Further note that the \texttt{func} object here is passed explicitly, but this would not be necessary as the default is to use the function that was last created.
When \texttt{x} and \texttt{y} are multiplied via \texttt{hlo\_multiply()}, the corresponding type inference function \texttt{stablehlo::infer\_types\_multiply()} is called, which checks that the operation is valid for the given input types, computes the output type of the operation, registers the operation in the body of the underlying StableHLO program, and then
returns a \texttt{FuncValue} object for the result.
Then, \texttt{hlo\_return()} sets the return values in the underlying \texttt{Func} object and returns the \texttt{Func} object itself, marking the end of the program creation.

Finally, the \texttt{repr()} generic is called on the \texttt{Func} object to get the string representation of the program.
This is implemented by recursively calling the \texttt{repr()} generic on its sub-components.
Note that the printer for \texttt{Func} and \texttt{FuncValue} objects uses the \texttt{repr()} generic internally to print the sub-components.

While some details were omitted in the above example, this shows the core ability of the \CRANpkg{stablehlo} package to easily create StableHLO programs in R.
For some more details, see the StableHLO specification or \href{https://r-xla.github.io/stablehlo/articles/stablehlo.html}{this vignette}.

\paragraph{PJRT}\label{pjrt}

The \CRANpkg{pjrt} package is the interface to the PJRT C++ library.
The PJRT library itself is header-only and only defines the interface for the underlying execution engine.
A specific implementation of the interface (e.g.~for CPU or NVIDIA GPU) is accessed by downloading prebuilt shared libraries and loading them dynamically.
Core components of the PJRT interface are buffer allocation, program compilation, and program execution, but many other API endpoints exist.

A practical consequence of this plugin architecture is that GPU support becomes considerably easier to install than with traditional CUDA-based R packages.
The CUDA PJRT plugin is self-contained: it ships with the required CUDA runtime libraries, so the user only needs a compatible NVIDIA driver and does not have to install a matching CUDA toolkit on the host system.

Because \CRANpkg{pjrt} only requires a StableHLO program as input, it is not tied to \CRANpkg{anvl} as a frontend.
Models trained in other frameworks --- for example PyTorch via \texttt{torch.export}, JAX via \texttt{jax.export}, or TensorFlow via the StableHLO converter --- can be exported to StableHLO and then loaded and executed from R through \CRANpkg{pjrt}, opening up a path to deploy and run pre-trained models in R workflows without depending on the original Python toolchain.
The reverse direction is equally interesting: StableHLO programs produced by \CRANpkg{anvl} can in principle be exported and run by any other StableHLO-consuming framework or runtime.
Realizing both directions in practice will require extending the \CRANpkg{stablehlo} package to support programs with dynamic dimensions and the corresponding shape-refinement passes, since exported programs typically use shape polymorphism that has to be specialized down to concrete shapes before XLA compilation; this is not yet supported.

The same dependency scale is also the reason why \CRANpkg{pjrt} ships prebuilt plugins rather than building them from source at install time.
A PJRT plugin transitively depends on LLVM, MLIR, the StableHLO and XLA dialects, parts of the XLA runtime, and further libraries such as protobuf, Abseil, and Eigen, adding up to a rather complicated Bazel-driven build that produces gigabytes of artifacts.
On the GitHub Actions runners we use for CI, a full plugin build takes between one and four hours, so expecting end users to run this as part of a source install would be unreasonable.
In addition, separate plugins are built for CPU and CUDA today (with Metal likely to follow), each targeting several operating systems, so building from source on the user's machine would mean re-running this process for every platform-backend combination.

Just like for the \CRANpkg{stablehlo} package, we only show the core functionality of \CRANpkg{pjrt} here and refer to the corresponding \href{https://r-xla.github.io/pjrt/articles/pjrt.html}{vignette} for more details.

As a first step, we will compile the StableHLO program from above.
For this, the R \texttt{character(1)} first needs to be converted into a \texttt{PJRTProgram} object.

\begin{verbatim}
library(pjrt)
program <- pjrt_program(src = hlo_string, format = "mlir")
\end{verbatim}

Then, this \texttt{PJRTProgram} can be compiled into a \texttt{PJRTLoadedExecutable} object, targeting a specific device.
A \texttt{PJRTDevice} is obtained from a \texttt{PJRTClient}, which in turn wraps a \texttt{PJRTPlugin} --- the dynamic library implementing the PJRT interface for a given hardware backend.
Each platform has a single cached client, constructed on demand from a string identifier such as \texttt{"cpu"} or \texttt{"cuda"}.

\begin{verbatim}
device <- pjrt_device("cpu")
executable <- pjrt_compile(program, device = device)
executable
\end{verbatim}

\begin{verbatim}
#> <PJRTLoadedExecutable>
\end{verbatim}

In order to execute this program, \texttt{PJRTBuffer} objects need to be created that are multi-dimensional arrays.
Below, we create two simple \texttt{f32} arrays of shape \texttt{2x2} and specify the device to be the CPU, which will use the buffer allocation implementation of the PJRT CPU plugin.
For CPU, the distinction between device and client is not so important as there is usually only one CPU, so there is a one-to-one correspondence.
When working with GPUs, there might be multiple GPUs available and the device represents one of them (by default the first one), and it matters on which GPU the buffer will be allocated.
However, multi-GPU setups are not properly supported in \CRANpkg{pjrt} yet, but this could be added in the future.

\begin{verbatim}
x <- pjrt_buffer(1:4, dtype = "f32", device = "cpu")
x
\end{verbatim}

\begin{verbatim}
#> PJRTBuffer 
#>  1
#>  2
#>  3
#>  4
#> [ CPUf32{4} ]
\end{verbatim}

\begin{verbatim}
y <- pjrt_buffer(5:8, dtype = "f32", device = "cpu")
y
\end{verbatim}

\begin{verbatim}
#> PJRTBuffer 
#>  5
#>  6
#>  7
#>  8
#> [ CPUf32{4} ]
\end{verbatim}

Various extractors exist for \texttt{PJRTBuffer} objects, for example:

\begin{verbatim}
shape(x)
\end{verbatim}

\begin{verbatim}
#> [1] 4
\end{verbatim}

\begin{verbatim}
device(x)
\end{verbatim}

\begin{verbatim}
#> <CpuDevice(id=0)>
\end{verbatim}

We can multiply the two buffers with each other by passing them to the \texttt{pjrt\_execute()} function along with the executable.

\begin{verbatim}
out <- pjrt_execute(executable, x, y)
out
\end{verbatim}

\begin{verbatim}
#> PJRTBuffer 
#>   5 12
#>  21 32
#> [ CPUf32{2x2} ]
\end{verbatim}

Conversion back to R is possible via the \texttt{as\_array()} function.

\begin{verbatim}
as_array(out)
\end{verbatim}

\begin{verbatim}
#>      [,1] [,2]
#> [1,]    5   12
#> [2,]   21   32
\end{verbatim}

A noteworthy aspect of PJRT is that its execution model is fundamentally asynchronous: \texttt{pjrt\_execute()} returns immediately with \texttt{PJRTBuffer} objects whose contents may not yet be available, and dependencies between successive calls are tracked internally so that subsequent operations can be enqueued without waiting for previous ones to finish.
The \CRANpkg{pjrt} package surfaces this via \texttt{is\_ready()} for non-blocking status checks, \texttt{await()} to block until a buffer is ready, and \texttt{as\_array\_async()} together with \texttt{value()} to overlap device-to-host transfers with other host-side work.
This makes it possible to pipeline computation, communication, and host-side R code, which is particularly valuable on accelerators where launch latency would otherwise dominate.
Conversion back to R via \texttt{as\_array()} implicitly synchronizes, since the result has to be materialized.
This async execution model carries through to \CRANpkg{anvl}: because \texttt{AnvlArray}s on the XLA backend wrap \texttt{PJRTBuffer}s, calls to \texttt{jit()}-compiled \CRANpkg{anvl} functions return immediately with \texttt{AnvlArray}s whose contents may not yet be ready, and the same \texttt{is\_ready()}, \texttt{await()}, and \texttt{as\_array\_async()} interface applies.

Because \texttt{PJRTBuffer} objects are external pointers, custom (de)serialization methods are needed.
For this, we implement the safetensors format \citep{ref-safetensors}.

\begin{verbatim}

tmp <- tempfile(fileext = ".safetensors")
safetensors::safe_save_file(list(x = out), tmp, framework = "pjrt")
reloaded <- safetensors::safe_load_file(tmp, framework = "pjrt")
reloaded$x
\end{verbatim}

\begin{verbatim}
#> PJRTBuffer 
#>   5 12
#>  21 32
#> [ CPUf32{2x2} ]
\end{verbatim}

The \CRANpkg{quickr} package serves as a second, experimental, backend for \CRANpkg{anvl} and was independently developed.
It transpiles R code to Fortran code and compiles it.
Below, we can recreate the StableHLO program from above.
One semantic difference is that it uses data type \texttt{f64} instead of \texttt{f32} because \CRANpkg{quickr} operates directly on R as opposed to \CRANpkg{pjrt} which defines its own array type.

    type(y = double(NA, NA))
  )
  x * y
}

  matrix(c(1, 2, 3, 4), nrow = 2),

)
\end{verbatim}

\begin{verbatim}

#> [1,]    5   21
#> [2,]   12   32
\end{verbatim}

There are also various other differences with respect to the underlying implementation.
Instead of offering a builder API like in \CRANpkg{stablehlo}, \CRANpkg{quickr} works via static analysis of the R code.
Furthermore, the argument types of the inputs need to be specified via the \texttt{declare()} function.
Here, \texttt{double(NA,\ NA)} means a 2-dimensional array with unknown dimensions, but concrete dimensions can also be specified.
One advantage of \CRANpkg{quickr} over \CRANpkg{pjrt} and \CRANpkg{stablehlo} is that the former does not necessitate recompilation for every new input type combination (which includes the shapes).
A disadvantage is that \CRANpkg{quickr} currently does not support GPU acceleration and also less data types.

We stress that the \CRANpkg{quickr} backend is currently experimental and not yet usable in practice: only a small subset of \CRANpkg{anvl}'s primitives have a \texttt{quickr} lowering rule, and the backend does not yet exploit \CRANpkg{quickr}'s native support for shape dynamism, so today it offers no real advantage over the XLA backend.
The long-term goal of the \CRANpkg{quickr} backend is to enable ahead-of-time (AOT) compilation with proper shape dynamism, so that a compiled program can be reused across a wide range of input shapes without re-tracing or recompilation.
Realizing this requires reworking the \texttt{AnvlGraph} intermediate representation so that its shape datatypes can carry \emph{symbolic} shapes rather than only fully static ones, which is a substantial change to the IR and is still work in progress.

\subsubsection{Automatic Differentiation}\label{automatic-differentiation}

Besides JIT compilation, another core feature of \CRANpkg{anvl} is its ability to automatically compute derivatives of R functions.
Here, we briefly cover some central ideas behind AD to provide the background for understanding \CRANpkg{anvl}'s design.
For a comprehensive treatment, we refer the reader to \citet{ref-griewank2008evaluating}.

\paragraph{Evaluation Procedures and the Chain Rule}\label{evaluation-procedures-and-the-chain-rule}

AD exploits the fact that every computer program, no matter how complex, executes a sequence of elementary operations whose derivatives are known.
Following \citet{ref-griewank2008evaluating}, consider a function \(F : \R^n \to \R^m\) that is evaluated by a computer program.
Any such evaluation can be decomposed into a sequence of elementary operations (called \emph{elemental functions}) \(\varphi_i\) such as addition, multiplication, \(\sin\), \(\exp\), etc.
The evaluation produces a sequence of intermediate variables

\[v_i = \varphi_i(v_j)_{j \prec i}, \qquad i = 1, \ldots, l\]

where \(v_{1-n}, \ldots, v_0\) are the input variables (i.e., \(x\)), \(v_1, \ldots, v_l\) are the intermediate variables, and \(v_{l-m+1}, \ldots, v_l\) are the output variables (i.e., \(y\)).
Here, the notation \(j \prec i\) indicates that \(v_j\) is a direct argument of \(\varphi_i\).
This sequence of assignments is called an \emph{evaluation procedure} and can be visualized as a directed acyclic graph (the \emph{computational graph}), where each node corresponds to an intermediate variable and edges represent data dependencies.

Since each elemental function \(\varphi_i\) is differentiable, the chain rule can be applied systematically to compute derivatives of \(F\).
The two main modes of AD differ in the \emph{order} in which the chain rule is evaluated, leading to different computational trade-offs.

\paragraph{Forward Mode}\label{forward-mode}

Forward-mode AD propagates derivatives alongside the function evaluation, computing a Jacobian-vector product \(\dot{y} = F'(x) \dot{x}\) in a single forward sweep.
Obtaining the full Jacobian \(F'(x) \in \R^{m \times n}\) requires \(n\) such sweeps, which makes forward mode efficient when \(n\) is small relative to \(m\).
\CRANpkg{anvl} currently only implements reverse mode, which we describe next.

\paragraph{Reverse Mode}\label{reverse-mode}

Reverse-mode AD (also called the \emph{adjoint} mode) propagates sensitivities backward through the evaluation procedure.
Given an adjoint vector \(\bar{y} \in \R^m\), it computes for each intermediate variable \(v_i\) the corresponding adjoint

\[\bar{v}_i = \sum_{k \succ i} \bar{v}_k \fp{v_k}{v_i}\]

proceeding in reverse order, from outputs to inputs, where \(k \succ i\) denotes that \(v_i\) is a direct argument of \(\varphi_k\).
The final result \(\bar{x} = \bar{y}^\top F'(x)\) is a \emph{vector-Jacobian product} (VJP).

The key advantage of reverse mode is that for a scalar-valued function (\(m = 1\)), setting \(\bar{y} = 1\) yields the full gradient \(\nabla F(x) \in \R^n\) in a \emph{single} reverse sweep, regardless of \(n\).
This result is sometimes referred to as the ``cheap gradient principle'': the computational cost of evaluating the gradient is bounded by a small constant multiple (typically 2--5) of the cost of evaluating \(F\) itself (\citet{ref-griewank2008evaluating}, Chapter 3).
This makes reverse mode dramatically more efficient than forward mode for the common case of gradient-based optimization, where one differentiates a scalar loss with respect to potentially many parameters.

The trade-off is that reverse mode requires access to the intermediate values \(v_i\) from the forward evaluation during the backward sweep, since they are needed to evaluate the local partial derivatives \(\fp{v_k}{v_i}\).
This introduces additional memory overhead that scales with the depth of the computation.

In \CRANpkg{anvl}, we implement backward-mode AD as a graph-to-graph transformation, as described in the next section.
Each primitive operation stores its own local VJP rule, and the transformation walks the computational graph in reverse topological order to accumulate gradients.

\subsection{Package Design and Features}\label{package-design-and-features}

Having coverved the underlying basics, we will now cover the main design of the \CRANpkg{anvl} frontend.
The core idea of the package is centered around code transformations, an approach adopted from JAX \citep{ref-jax}.
In general, there are three types of transformations:

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  Transforming R code into a computational graph via tracing.
\item
  Transforming a computational graph into another computational graph (e.g.~for automatic differentiation).
\item
  Transforming a computational graph into an executable (either via \CRANpkg{stablehlo} and \CRANpkg{pjrt} or \CRANpkg{quickr}).
\end{enumerate}

We will now cover these three types of transformations in more detail.
Then, we explain how these transformations are wrapped in the main user interface.

\subsubsection{Code transformations}\label{code-transformations}

\paragraph{Tracing R code into a computational graph}\label{tracing-r-code-into-a-computational-graph}

When it comes to translating code into other code, there are in principle two approaches:

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  Static analysis of the code.
\item
  Dynamically executing the code with ``special values'' that record certain operations and build up a (partial) representation of the program.
\end{enumerate}

In \CRANpkg{anvl}, we are following the latter approach, because it allows to seemlessly combine standard R code with jit-compilable operations.
The function that takes in an R function and returns a computational graph is called \texttt{trace\_fn()}.
In order for this to be useful, the function needs to call into anvl primitive functions.
These primitive functions are similar to the stableHLO operations like \texttt{hlo\_add} from earlier.

In the \CRANpkg{anvl} package these are called \texttt{nvl\_\textless{}op\textgreater{}} functions.
Howveer, they are not exported directly, but in thin wrappers like \texttt{nv\_add}, which perform some convenience transformations (like type casting) that make them more ergonomic to use.
These thin wrappers call into the primitive functions, however.
The primitive functions are essentially graph building functions, also similar to the \texttt{hlo\_add} function from earlier.
Below, we create an R function that takes in two values and multiplies them.
Note that we could have also used the \texttt{*} infix operator, but we want to illustrate the use of the \texttt{nv\_multiply} function.

\begin{verbatim}
library(anvl)
multiply_r <- function(x, y) {
  nv_mul(x, y)
}
\end{verbatim}

We can now trace this function into a computational graph.
At the beginning, this will create a \texttt{anvl::GraphDescriptor} that primitive functions can attach to to build up the graph.
The types for the inputs are inferred from the example arguments.
The \texttt{trace\_fn} transformation uses these example inputs to obtain abstract \texttt{GraphBox} objects, which are passed around during the evaluation of the function.
They essentially define the type (shape and data type), and uniquely identify a specific \texttt{GraphValue} within a specific \texttt{GraphDescriptor}.
Note that \texttt{trace\_fn} also takes in non-array values, which are evaluated like normal R code.

\begin{verbatim}
graph <- trace_fn(multiply_r, list(x = nv_scalar(1, "f32"), y = nv_scalar(5, "f32")))
graph
\end{verbatim}

\begin{verbatim}
#> <AnvlGraph>
#>   Inputs:
#>     %x1: f32[]
#>     %x2: f32[]
#>   Body:
#>     %1: f32[] = mul(%x1, %x2)
#>   Outputs:
#>     %1: f32[]
\end{verbatim}

Once the evaluation of \texttt{multiply\_r} with the special \texttt{GraphBox} objects is complete, the \texttt{GraphDescriptor} is converted into a \texttt{Graph} object.
The only difference between them is that \texttt{GraphDescriptor} does some book-keeping during program creation which is not needed anymore afterwards.

From the printed output of the graph we can see that a graph has:

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  Inputs: \texttt{GraphValue}s that are the inputs to the graph.
\item
  Constants: \texttt{GraphValue}s that are constants in the graph.
\item
  Body: A list of \texttt{PrimitiveCall} objects, which represent the primitive operations that make up the graph.
\item
  Outputs: \texttt{GraphValue}s that are the outputs of the graph.
\item
  In-tree and Out-tree: Because \texttt{Graph}s only take in array values, but we want to support nested data strctures, the in-tree and out-tree define how to (un-)flatten the inputs and outputs of the graph.
\end{enumerate}

\paragraph{Transforming a Graph into another Graph}\label{transforming-a-graph-into-another-graph}

Once our program no longer consists arbitrary R code, but is in a standardized form, we can transform the program by creating another \texttt{Graph} object.
Currently, the only example for such a type of transformation is the gradient transformation, which is implemented in the \texttt{transform\_gradient()} function.
It takes in a \texttt{Graph} and the names of the arguments to compute the gradient with respect to.

\begin{verbatim}
gradient_graph <- transform_gradient(graph, wrt = c("x", "y"))
\end{verbatim}

The \CRANpkg{anvl} package does not put any restrictions on \emph{how} such a transformation is implemented.
Usually, however, such a transformation needs to store information for how to transform each primitive.
For this reason, the \texttt{AnvlPrimitive} objects have transformation rules attached to them, accessed via \texttt{{[}{[}\textless{}rule\textgreater{}{]}{]}}.
For example, the reverse rule for the addition primitive is implemented as follows:

\begin{verbatim}
prim_add[["reverse"]]
\end{verbatim}

\begin{verbatim}
#> function (inputs, outputs, grads, .required) 
#> {
#>     grad <- grads[[1L]]
#>     list(if (.required[[1L]]) grad, if (.required[[2L]]) grad)
#> }
#> <environment: namespace:anvl>
\end{verbatim}

The transformation function \texttt{transform\_gradient} basically walks the graph backwards and applies the reverse rules to each primitive, producing a \texttt{Graph} that contains the forward and backward pass.

\paragraph{The Gradient Transformation}\label{the-gradient-transformation}

The \texttt{transform\_gradient()} function implements backward-mode automatic differentiation (also known as reverse-mode AD).
Given a \texttt{Graph} representing a function \(f : \mathbb{R}^n \to \mathbb{R}\), it produces a new \texttt{Graph} that computes both the original output and the gradients of the output with respect to specified inputs.

More concretely, consider a computational graph as a sequence of primitive operations \(v_k = g_k(v_{\text{pa}(k)})\), where \(v_{\text{pa}(k)}\) denotes the parent values (inputs) of operation \(k\).
The goal is to compute \(\bar{v}_i = \frac{\partial f}{\partial v_i}\) for each intermediate value \(v_i\).
Starting from \(\bar{f} = 1\), the backward pass walks the graph in reverse topological order and accumulates gradients via the chain rule:
\[\bar{v}_i = \sum_{k : i \in \text{pa}(k)} \bar{v}_k \cdot \frac{\partial g_k}{\partial v_i}\]
Each primitive's reverse rule computes the local partial derivatives \(\frac{\partial g_k}{\partial v_i}\) and multiplies them with the incoming gradient \(\bar{v}_k\).
This is the standard vector-Jacobian product (VJP) formulation of reverse-mode AD, which avoids materializing the full Jacobian matrices of intermediate operations.
Instead, each reverse rule directly computes the product of the incoming gradient vector with the local Jacobian, which is typically much cheaper and can often be expressed in closed form.

In the implementation, the transformation walks the graph in reverse order.
For each \texttt{PrimitiveCall}, it looks up the corresponding reverse rule stored in the primitive's \texttt{@rules{[}{[}"reverse"{]}{]}} field.
This reverse rule receives the original inputs and outputs of the primitive call (from the forward pass) as well as the incoming gradient \(\bar{v}_k\), and returns the gradient contributions for each input.
These contributions are accumulated across all uses of a value to obtain the total gradient \(\bar{v}_i\).

Because the reverse rules are themselves expressed in terms of primitive operations (i.e., they call \texttt{nvl\_*} functions), the gradient graph is built using the same tracing mechanism as the forward pass.
This means that the gradient computation is itself a \texttt{Graph} that can be further transformed or compiled --- for example, one could in principle compute higher-order derivatives by applying \texttt{transform\_gradient()} again to the gradient graph.

An important design choice is that the reverse rules are stored per-primitive in the \texttt{@rules} field, rather than being hard-coded in the transformation.
This makes it straightforward to add differentiation support for new primitives (see Section \ref{extending}) and keeps the transformation logic generic.

After this is done, the resulting gradient graph is inlined into the parent \texttt{GraphDescriptor}.

\paragraph{JIT vs.~Eager Mode}\label{jit-vs.-eager-mode}

In addition to the JIT-compiled execution path described above, \CRANpkg{anvl} also supports an \emph{eager} execution mode in which primitive operations are dispatched and executed immediately rather than being recorded into a \texttt{Graph}.
Concretely, calling \texttt{nv\_add(1,\ nv\_array(1:2))} outside of any tracing context performs the addition right away and returns an \texttt{AnvlArray}, instead of merely registering a \texttt{PrimitiveCall} on a \texttt{GraphDescriptor}.
Eager mode was introduced for two reasons.
First, it makes programs significantly easier to debug: intermediate values can be inspected directly with R tooling (e.g., \texttt{print()}, \texttt{browser()}), without having to disable JIT compilation or insert auxiliary \texttt{nv\_print()} calls into a traced program.
Second, it removes the need to maintain two versions of every user-defined function --- one written in plain R for interactive use or testing, and another wrapped in \texttt{jit()} for execution.
With eager mode, the same R function can be called directly during development and only wrapped in \texttt{jit()} when the user is ready to compile it, since both modes share the same semantics for the underlying \texttt{nv\_*} operations.
Eager mode is implemented as a special case of the same machinery used for tracing: when a primitive is invoked outside of any active \texttt{GraphDescriptor}, it falls back to executing immediately on the currently selected backend.

\subsubsection{Limitations of XLA}\label{limitations-of-xla}

While the JIT-compilation approach is very powerful, it inherits a few restrictions from the XLA compiler that shape how programs must be written.

The most prominent one is that all shapes must be known at compile time.
While the StableHLO language itself does support dynamic shapes, the XLA compiler requires them to be fully static, which also allows it to apply more aggressive optimizations.
As a consequence, operations whose output shape depends on the values of the inputs, such as \texttt{unique()} or boolean masking, cannot be expressed directly.
A common workaround is to pad inputs to a fixed maximum size and mask out the irrelevant entries with e.g.~\texttt{nv\_where()}.
This preserves static shapes at the cost of some wasted computation.
On GPUs this extra work is often acceptable because idle lanes are cheap, but on CPUs padding tends to hurt performance much more noticeably.

A related restriction is that StableHLO operations are designed so that every precondition can be verified at compile time, rather than producing runtime errors.
For example, subsetting-like operations such as \texttt{nvl\_dynamic\_slice()} silently clamp the slice indices to the valid range instead of raising an out-of-bounds error, and in \texttt{nvl\_gather()} out-of-bounds indices yield an implementation-defined result.
This means that any argument whose value must be validated (e.g.~an index vector that must lie within a given range) effectively has to be a static argument to the jitted function, so that \CRANpkg{anvl} can check it during tracing rather than relying on the compiled program to do so.
In practice, this is usually not a severe restriction, but it is something users need to be aware of when designing their programs.

\subsection{Extending}\label{extending}

One of the key design goals of \CRANpkg{anvl} is to be extendable.
It is possible to add new primitive operations, transformations, and even backends.

\subsubsection{New Primitive Operations}\label{new-primitive-operations}

The simplest way to extend \CRANpkg{anvl} is to add new primitive operations via \texttt{new\_primitive()}.
A primitive bundles an abstract-evaluation rule (mapping input shapes and dtypes to output shapes and dtypes) with a set of per-transformation rules --- a StableHLO lowering rule for the XLA backend, a \texttt{quickr} lowering rule for the Fortran backend, and a reverse-mode rule for automatic differentiation.
Any operation that can be expressed in terms of StableHLO and \CRANpkg{quickr} primitives can be added this way, and the same mechanism is used for all primitives shipped with \CRANpkg{anvl} itself.
For operations that cannot be expressed in StableHLO --- for example when wrapping a hand-written CUDA kernel or an existing C/C++ routine --- the XLA backend additionally exposes PJRT's custom-call FFI mechanism via \texttt{pjrt::pjrt\_register\_custom\_call()}, which lets users dispatch to a C/C++ handler from inside a compiled program (with separate CPU and CUDA implementations selected per PJRT platform).
A worked end-to-end example, including how to write the rules and register a custom call, is given in the \href{https://r-xla.github.io/anvl/articles/extending_primitive.html}{extending primitives vignette}.

\subsubsection{New Transformations}\label{new-transformations}

Because the \texttt{Graph} representation is a simple data structure consisting of inputs, outputs, and a sequence of \texttt{PrimitiveCall} objects, it is straightforward to implement new graph-to-graph transformations.
The \texttt{transform\_gradient()} function, which implements backward-mode automatic differentiation, is the primary example of such a transformation.
It walks the graph in reverse order and applies the backward rule stored in each primitive's \texttt{@rules{[}{[}"backward"{]}{]}} field to accumulate gradients.

Implementing a new transformation follows the same pattern: iterate over the \texttt{PrimitiveCall} objects in the graph, look up a corresponding rule for each primitive, and use these rules to construct a new graph.
For example, one could implement forward-mode automatic differentiation by walking the graph \emph{forward} and propagating Jacobian-vector products, or implement a graph optimization pass that fuses or simplifies sequences of primitive calls.
The key design decision is that the \CRANpkg{anvl} package does not impose any constraints on how a transformation is implemented --- it only provides the \texttt{Primitive} rule mechanism as a convenient way to store per-operation information.

\subsubsection{Custom Backends}\label{custom-backends}

Backends are represented by \texttt{AnvlBackend} objects that bundle the operations \CRANpkg{anvl} needs in order to treat a storage-and-execution mechanism as a target.
In essence, an \texttt{AnvlBackend} is a small collection of functions covering data construction (\texttt{new\_data}), metadata accessors (\texttt{dtype}, \texttt{shape}, \texttt{ambiguous}), conversions (\texttt{as\_array}, \texttt{as\_raw}), device handling (\texttt{platform}, \texttt{device}, \texttt{new\_device}), printing (\texttt{print\_data}), and JIT compilation (\texttt{jit}).
Each backend is registered under a string name and can be selected via \texttt{local\_backend()} or \texttt{with\_backend()}.
Two backends are currently shipped with \CRANpkg{anvl}: the default \texttt{"xla"} backend, which wraps \texttt{PJRTBuffer} objects and lowers \texttt{Graph}s to StableHLO, and the \texttt{"quickr"} backend described earlier, which stores plain R arrays and compiles \texttt{Graph}s to R code via \CRANpkg{quickr}.
Adding a new backend consists of implementing the \texttt{AnvlBackend} interface and, for each primitive that the backend supports, providing a lowering rule under a new name in the primitive's \texttt{@rules} field.

Conceptually, the \texttt{AnvlBackend} interface plays the role of a ``mini PJRT'': it captures the essential notions that any backend must provide --- how to hold array data, how to describe it, how to move it around, and how to compile and execute a graph --- but avoids the complexity of the full PJRT C API.
This makes it considerably easier to plug in new execution strategies that do not fit the PJRT model.
For example, a pure-R backend that simply interprets \texttt{Graph}s using base-R array operations can be implemented as an \texttt{AnvlBackend} without touching a compiler toolchain or a C++ plugin.

That said, the \texttt{AnvlBackend} interface is still work in progress and not yet fully specified.
Its precise shape will continue to evolve as we gain experience with additional backends, so authors of custom backends should currently expect occasional breaking changes.
A more streamlined and well-documented interface for registering custom backends is an area we plan to improve.

\subsection{Testing}\label{testing}

Ensuring correctness of both the forward evaluation and the gradient computation is critical for a code transformation framework.
In \CRANpkg{anvl}, we bootstrap correctness by comparing against the \CRANpkg{torch} package wherever possible.

For forward-pass testing, each primitive's JIT-compiled output is compared against the corresponding \CRANpkg{torch} function on randomly generated inputs.
For example, to test \texttt{nvl\_add}, we generate random arrays, compute the result using both \texttt{jit(nvl\_add)(x,\ y)} and \texttt{torch::torch\_add(x,\ y)}, and assert that the outputs match up to numerical tolerance.
This is done using helper functions like \texttt{expect\_jit\_torch\_binary()} and \texttt{expect\_jit\_torch\_unary()}, which handle the data generation and comparison.

For backward-pass testing, the same strategy is applied to gradient computation.
The anvl gradient (obtained via \texttt{jit(gradient(f))(x)}) is compared against the gradient computed by \CRANpkg{torch}'s autograd engine.
For non-scalar functions, both sides are wrapped with a summation to produce a scalar loss before differentiating, ensuring a consistent comparison.

Because \CRANpkg{torch} is not a hard dependency of the package, the torch-based tests are stored in \texttt{inst/extra-tests/} and are conditionally sourced only when \CRANpkg{torch} is installed.
This keeps the test suite lightweight for users who do not have \CRANpkg{torch}, while still running the full comparison suite in CI.
For primitives that have no direct \CRANpkg{torch} equivalent (e.g., random number generation or bitwise operations), manual tests with hardcoded expected values are used instead.

\subsection{Discussion}\label{discussion}

In this paper, we introduced the \CRANpkg{anvl} package, which is a code transformation framework for the R language.
Execution is handled via JIT compilation through either the OpenXLA compiler framework or by transpiling to Fortran via \CRANpkg{quickr}.
When using the OpenXLA backend, it is possible to make use of accelerators such as GPUs.
Furthermore, the package supports first-order automatic differentiation.

The biggest limitations of the current approach are twofold:

\begin{enumerate}
\def\labelenumi{\alph{enumi})}
\tightlist
\item
  Programs need to have static shapes, which means that operations such as \texttt{unique()} are currently not supported.
\item
  For the OpenXLA backend, programs need to be re-compiled for each new unique input type combination.
\end{enumerate}

\subsubsection{Future Work}\label{future-work}

We group the directions for future work into five categories: lifting current limitations and improving the IR, filling gaps in the API, adding more code transformations, supporting more compute platforms, and building downstream applications on top of \CRANpkg{anvl}.

\paragraph{Lifting Current Limitations and IR Improvements}\label{lifting-current-limitations-and-ir-improvements}

Most of the limitations discussed above stem from the design of the intermediate \texttt{AnvlGraph} and from XLA-imposed restrictions on the StableHLO programs we emit.
The most pressing direction is to extend the IR datatypes underlying \texttt{AnvlGraph} (in particular \texttt{AbstractArray}) to carry symbolic shapes rather than only fully static ones, so that the same graph can describe a family of input shapes.
This is a prerequisite for either of the two routes that would lift the static-shape and recompilation restrictions.
The first is to swap out the XLA compiler for the IREE compiler \citep{ref-iree}, which also consumes StableHLO programs but natively supports dynamic shapes; the downside is that IREE is considerably less mature than XLA and not yet a practical drop-in.
The second is to extend the \CRANpkg{quickr} backend, which targets Fortran and does support shape dynamism, so that it covers a larger fraction of \CRANpkg{anvl}'s primitives and would lift the second restriction as well (at least on CPU); today this backend is not yet far enough along to serve as such a replacement.
A related improvement is to decouple \texttt{AnvlGraph} from StableHLO more thoroughly: currently the IR is tied too closely to StableHLO, which limits the ability to target alternative backends and to represent higher-level optimizations that do not map directly to StableHLO concepts.

\paragraph{Filling Gaps in the API}\label{filling-gaps-in-the-api}

The set of primitive operations currently supported by \CRANpkg{anvl} and \CRANpkg{stablehlo} is rather narrow and is the simplest aspect of the package to extend.
On top of new primitives, additional API functions can be added, in particular focusing on linear algebra operations that are common in statistical and machine learning workloads.

\paragraph{Adding More Code Transformations}\label{adding-more-code-transformations}

So far, the only graph-to-graph transformation \CRANpkg{anvl} ships is reverse-mode automatic differentiation.
Several other transformations would substantially extend the package's capabilities: forward-mode automatic differentiation, which together with reverse mode enables efficient computation of higher-order derivatives such as Jacobians and Hessians; improved vectorization transformations in the spirit of JAX's \texttt{vmap}; and JET-style static analyzers that walk an \texttt{AnvlGraph} to catch shape, dtype, or other correctness issues before compilation.

\paragraph{Supporting More Compute Platforms}\label{supporting-more-compute-platforms}

On the hardware side, the set of supported PJRT plugins could be expanded.
The ZML project \citep{ref-zml} ships PJRT plugins for ROCm (AMD GPUs) and CUDA on Arm, and adopting these would extend \CRANpkg{anvl}'s reach to these platforms with comparatively little effort.
Proper support for Apple Silicon GPUs is more uncertain: the official \texttt{jax-metal} plugin appears to no longer be actively developed, but the experimental \href{https://github.com/tillahoffmann/jax-mps}{jax-mps} project explores routing JAX through Metal Performance Shaders, and if it matures it could provide a path to first-class Mac GPU execution for \CRANpkg{anvl} as well.
The main blocker for supporting additional platforms is not integration effort but testing infrastructure: each backend needs continuous-integration runners with the corresponding hardware in order to keep the test suite green, and access to such runners (especially for AMD GPUs, Arm, and Apple Silicon) is currently the limiting factor.

\paragraph{Downstream Applications}\label{downstream-applications}

\CRANpkg{anvl} is designed as low-level infrastructure on which more specialized libraries can be built.
We see several high-impact directions:

\begin{itemize}
\tightlist
\item
  A Bayesian inference library along the lines of NumPyro \citep{ref-numpyro} or BlackJAX \citep{ref-blackjax}, including HMC and variational inference algorithms that benefit directly from JIT compilation, GPU acceleration, and AD.
\item
  A deep learning library exposing high-level neural network modules and gradient-based optimizers, analogous to the Flax/Optax stack in the JAX ecosystem.
\item
  Differential equation solvers (analogous to Diffrax), which combine naturally with AD for gradient-based fitting of dynamical systems.
\item
  Reimplementing statistical methods that already exist in R but do not scale to large datasets because their current implementations are too slow, where JIT compilation, GPU acceleration, and AD could yield substantial speedups while keeping the implementation in pure R.
\end{itemize}
