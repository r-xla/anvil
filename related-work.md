# Related work: R packages for compilation, autodiff, and GPU

**Anvil occupies an empty niche in R**: no existing CRAN or GitHub package provides the JAX-style triad of composable JIT compilation, source-to-source automatic differentiation, and first-class GPU execution on native R code. Anvil (r-xla/anvil) is an experimental, MIT-licensed package (v0.1.0.9000, not on CRAN as of April 2026) that lowers traced R functions — operating on typed `AnvilTensor`/`AnvilArray` values — into StableHLO programs executed via OpenXLA's PJRT runtime (through sibling packages `{pjrt}`, `{stablehlo}`), with an experimental `{quickr}`-based Fortran CPU backend. Reverse-mode AD is limited to scalar-output functions, shapes are fixed per compilation, and the project is explicitly framed as an R analogue of JAX's autodidax tutorial and the microjax project.

Prior art in R clusters into four roughly disjoint camps. **(1) C++ FFI frameworks** (Rcpp and its derivatives) are production infrastructure but compile user-authored C++, not R. **(2) Statistical AD toolchains** (TMB, RTMB, nimble, Stan Math) deliver excellent autodiff — often higher-order, often with sparse-linear-algebra support — but are CPU-only and scoped to likelihood/Bayesian inference. **(3) Deep-learning bindings** (torch, tensorflow, keras3, greta) offer GPU autodiff but either require rewriting code in a tensor DSL (torch), or delegate everything to Python via reticulate (tensorflow, keras3, greta). **(4) Domain-specific R compilers** (odin→C, dust→C++/CUDA, gpuMagic→OpenCL, quickr→Fortran) demonstrate the transpile-a-subset-of-R pattern but each lacks one of anvil's pillars. **Two historical LLVM-based projects** (Temple Lang's Rllvm/RLLVMCompile and Neal's pqR fork — which uniquely includes preliminary native AD) attempted general R compilation and effectively stalled, underscoring both the ambition and the maintenance risk of anvil's approach.

The table below summarizes the ~20 most directly comparable packages. Rows are grouped by primary focus, and within each row the **Compilation / Autodiff / GPU** columns capture whether the package provides that capability and how.

## What this landscape tells us about anvil's contribution

Anvil is best read as **the missing intersection of three mature but disjoint R subcultures**: the statistical AD tradition (TMB, RTMB, nimble, Stan) has the algorithmic sophistication but no GPU; the deep-learning imports (torch, tensorflow, keras3) have GPU and autograd but require tensor-DSL rewrites or Python; and the domain-specific R compilers (odin, dust, gpuMagic, quickr) have demonstrated that transpiling a typed subset of R to a performance backend is both tractable and sustainable when scope is disciplined. Anvil inherits that discipline — traced R, shape-specialized compilation — while swapping the target from C/C++/Fortran/OpenCL to **XLA/StableHLO**, which uniquely provides autodiff, kernel fusion, and multi-device execution (CPU/GPU/TPU) behind a single IR.

**Two comparisons matter most** for a related-work narrative. Against **torch**, anvil trades tape-based eager execution for whole-function JIT and promises vendor-neutral hardware targeting, at the cost (today) of scalar-output-only AD and a non-production status. Against **RTMB / nimble**, anvil generalizes beyond likelihood-fitting into arbitrary differentiable numerical programming and gains a GPU path, while giving up higher-order AD, sparse-aware atomics, and a decade of stats-community ecosystem. The **historical lessons** from Rllvm/RLLVMCompile, pqR, Ra/jit, and the OpenCL chain (gpuR archived, gputools archived, clrng dependency-fragile) all argue that the decisive question for anvil is not technical feasibility — gpuMagic and quickr already prove R→hardware compilation works — but whether tying to a well-funded external IR stack (OpenXLA/PJRT) can finally carry a general-purpose R compiler past the adoption and maintenance cliff that swallowed every previous attempt.

## Conclusion

R has no shortage of specialized tools for any *one* of compilation, autodiff, or GPU, but the Cartesian product is essentially empty today: **only `torch` combines autodiff and GPU natively, only `dust` combines compilation and GPU, and no production package combines all three natively on R code**. Anvil's bet is that XLA collapses these three axes into one infrastructure, with the r-xla split into `pjrt` / `stablehlo` / `anvil` / `quickr` deliberately mirroring JAX's layering. Its current scalar-output, shape-rigid, experimental status is typical for a new compiler project; whether it becomes the first durable general-purpose R compiler or joins Rllvm and pqR in the archive of ambitious prototypes will depend primarily on scope discipline and ecosystem integration, not on the underlying XLA technology, which is already battle-tested outside R.
